# 项目开发规则 (Project Development Rules)

本文件包含本项目特定的开发规则和约束，供AI助手和开发人员参考。

## 做任何事情必须遵守的rules(常驻下上文内)
**所有回复必须使用中文。**
**冲突解决：先看优先级；同一级内，序号越小优先。**

### Rule 1 · 核心原则
- 不即兴发挥 —— 严格按任务与规则执行。
- 不过度设计 —— 只做当前必要工作。
- 不偏离流程 —— 每次任务均执行完整流程。

### Rule 2 · 任务前置
- 在开始写代码前，必须先分析任务目标，制定明确计划。
  - 说明需修改的模块/函数/组件。
  - 写清改动的原因、范围及预期效果。
  - 必须输出"计划草稿"作为 AI 的第一步交互。
- 严禁任务不明即编程。

### Rule 3 · 精准定位
- 仅修改与任务直接相关的文件与代码行。
  - 每个改动需写明理由。
  - 不得牵连无关文件或模块。
- 禁止擅自引入新抽象或结构性重构，除非明确要求。

### Rule 4 · 最小改动
- 所有代码更改必须是"任务所需的最小实现"：
  - 不得添加无关日志、注释、TODO、调试输出。
  - 不得"顺手"修复无关问题。
- 若发现其他问题，应标记 TODO，留待后续处理。

### Rule 5 · edit_file 操作
- 每次使用 `edit_file` 前，**必须完整 reload 源文件一次**，以确保上下文一致。
- 若未 reload，禁止发起修改指令。

### Rule 6 · 测试驱动开发（TDD）
- 在开发或重构模块前，必须优先编写测试用例，覆盖边界条件与关键逻辑。
- 未完成测试编写，不得提交实现代码。

### Rule 7 · 提交前检查
- 所有提交前，必须检查以下内容：
  - 是否符合任务计划与范围。
  - 是否正确实现目标。
  - 是否引入副作用或影响下游模块。
  - 是否符合代码风格，与现有代码一致。
- 不通过自检，不得继续。

### Rule 8 · 交付说明
- 提交后必须输出：
  - 所有修改的文件与具体内容摘要。
  - 修改原因说明。
  - 假设与风险说明。
- 该交付内容作为 review 依据，不可省略。

### Rule 9 · 后向兼容
- 无需向后兼容，可以自由破坏旧格式。前提是新的格式必须全面优于旧格式。

## 附录 A · 推荐工具与持续改进守则

| 工具/守则 | 作用 | 使用时机 |
|-----------|------|---------|
| **Sequential Thinking** | 拆解复杂任务、保持上下文一致 | 问题范围模糊、多方案探索 |
| **Context7** | 拉取最新官方文档、示例 | API 模糊、版本差异大 |
| **Continuous Learning & Business Value** | 跟踪技术趋势、平衡创新与稳定 | 定期回顾或用户另行要求 |

> 工具使用不得削弱 Rule 1–8 的任何硬约束。

## 附录 B · 优先级速查表

| 规则 | 关键动作 | 优先级 | 遗漏后果 |
|------|----------|--------|----------|
| **6** | 先写测试再写实现（TDD） | 高 | 不得提交实现 |
| **3** | 明确修改点与范围 | 高 | 改动错误 |
| **4** | 限定最小改动 | 高 | 引入冗余 |
